# 이 프로젝트에 대한 규칙

이 리포는 Next.js + Supabase 웹 개발을 위한 **범용 기술 스킬 모음집**이다. Claude Code 스킬(`skills/*/SKILL.md`)을 작성·수정할 때 아래 규칙을 항상 지킨다.

## 1. 화물로 클론 금지 (최우선 원칙)

화물로(화주/기사 매칭 화물 마켓플레이스, 사용자의 실제 서비스)는 **참고 대상일 뿐**이며, 스킬 내용에 화물로의 구체적 비즈니스 로직(화물 매칭, 입찰, 배차, 화주/기사라는 역할명 등)을 그대로 넣으면 안 된다. 스킬은 어떤 종류의 서비스에도 적용 가능한 일반화된 패턴(예: "역할이 N개인 마켓플레이스", "provider/requester")으로 작성한다. 화물로는 영감의 원천이지 템플릿이 아니다.

## 2. 로드맵은 "개발 단계" 기준 (도메인 기준 아님) — 2번째로 확정된 원칙

처음엔 인증/DB설계/결제/배포 같은 **기능 도메인** 단위로 스킬을 나눴으나, 이는 사용자의 의도와 맞지 않았다. 사용자가 원하는 건 어떤 기능을 만들든 (인증이든 결제든 채팅이든) 거쳐가는 **개발 단계**별 스킬이다: 기획 → 디자인 → 데이터 모델링 → 기능 구현 → 페이지 구현 → 렌더링 → 배포 → 마케팅/계측. 각 스킬은 "이 단계에서 무엇을 어떻게 결정/구현할지"를 다루며, 특정 기능 도메인에 묶이지 않는다. (도메인 기준으로 만들었던 구버전 2개 스킬은 `skills/_archived-v1-domain-based/`에 보관, 활성 로드맵에서는 제외.)

8단계 앞에는 0단계(`essential-feature-discovery`)가 있다. 1~8단계는 "이미 만들기로 한 기능 하나"를 다루지만, 0단계는 "이 서비스에 어떤 기능이 필요한가"를 먼저 정하는 단계로, 프로젝트당 한 번 실행하고 이후 새 기능 요청이 들어올 때마다 재참조한다. 새 단계 스킬을 만들 때 이 둘의 책임 경계(0단계=무엇을 만들지 정함, 1단계 이상=정해진 것을 어떻게 만들지 다룸)를 섞지 않는다.

## 3. 스킬 범위

- 작은 단위로 모듈화한다 (하나의 거대한 스킬 금지) — 단계별로 나누되 각 단계 스킬도 너무 비대해지지 않게 한다.
- 마켓플레이스 특화 로직(에스크로, 매칭 알고리즘, 입찰)은 스킬 범위 밖이다 — 이건 매번 서비스마다 다르므로 일반화 대상이 아니다.
- 작동하는 템플릿 코드 파일을 만들지 않는다. SKILL.md 안의 코드는 예시일 뿐이며, 실제 프로젝트의 코드는 Claude가 그때그때 생성한다.

## 4. 보안

화물로 프로젝트에서 Supabase `service_role` 키와 Toss 시크릿 키가 배포 스크립트(`final_deploy.command`, `SETUP.md`)에 평문으로 커밋된 사고가 있었다 (키는 이미 교체됨). 모든 스킬, 특히 데이터 모델링·기능 구현·배포 관련 스킬은 "민감한 키를 코드/문서/배포 스크립트에 평문으로 남기지 않는다"는 원칙을 항상 반영해야 한다.

## 5. 언어 규칙 (3번째로 확정된 원칙)

- **SKILL.md 본문**: 영문으로 작성한다.
- **그 외 모든 문서** (README.md, CHANGELOG.md, PROJECT_BRIEF.md, CLAUDE.md, evals.json 등)와 **채팅 답변/리뷰**: 한글로 작성한다.
- 이미 한글로 작성됐던 구버전 스킬(아카이브됨)은 굳이 영문으로 재작성하지 않는다 — 새로 만드는 스킬부터 적용.

## 6. 구조

- 스킬은 `skills/<skill-name>/SKILL.md`에 둔다.
- 새 스킬을 추가하거나 기존 스킬을 의미 있게 수정하면 `CHANGELOG.md`에 한 줄 남긴다.
- 전체 로드맵과 설계 배경은 `PROJECT_BRIEF.md` 참고.

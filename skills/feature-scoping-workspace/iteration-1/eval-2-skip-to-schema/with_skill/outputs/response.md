스키마부터 가시죠 — 그런데 그 전에 잠깐만요.

지금 바로 `ratings`/`reviews` 테이블 컬럼을 정의하면, 나중에 "이 필드가 왜 필요했더라", "여기 NOT NULL이어야 하나" 같은 질문이 구현 중간에 다시 튀어나올 가능성이 높습니다. 스키마 설계는 "이 기능이 뭘 해야 하는가"가 정해진 다음 단계 작업이라서요. 지금은 그 앞 단계인 스코핑이 빠져 있는 상태입니다 — 5분만 먼저 정리하고 가면, 컬럼 정의할 때 헷갈릴 일이 줄어듭니다.

몇 가지만 답해주시면 바로 스코프 문서로 정리해드릴게요.

1. **문제**: provider가 평점을 받는 기능을 왜 지금 추가하나요? 예를 들어 "requester가 예약 전에 provider를 신뢰할 수 있는지 판단할 방법이 없다" 같은 식으로요. 평점/리뷰가 해결하려는 진짜 문제가 뭔가요?

2. **행위자**: requester(평가를 남기는 사람)와 provider(평가를 받는 사람) 외에, admin이 리뷰를 검토/삭제하는 역할도 필요한가요, 아니면 이 두 역할만으로 충분한가요?

3. **플로우**: 예를 들면 — requester가 예약을 완료한 뒤 평점을 남길 수 있는 시점이 되고 → 별점+텍스트를 입력하고 → provider 프로필에 반영되는, 이런 흐름이 맞나요? provider가 리뷰에 답글을 달 수 있어야 하나요? 평가를 안 남기면 어떻게 되나요? 부적절한 리뷰는 신고/삭제가 가능해야 하나요?

4. **MVP 경계**: v1에 꼭 있어야 하는 것과, 나중으로 미뤄도 되는 것을 나눠볼게요. 예를 들어 별점 + 짧은 텍스트 리뷰는 v1, provider의 리뷰 답글 기능이나 리뷰에 대한 "도움이 됐어요" 투표는 나중으로 미룰 수 있을 것 같은데, 맞나요? 아니면 처음부터 다 필요한가요?

5. **성공 기준**: 이 기능이 잘 작동했다는 걸 어떻게 확인할 수 있을까요? 예를 들어 "예약 완료 후 N일 내 평점 작성률 X% 이상" 같은 식으로요.

이 답변들이 모이면 `ratings`/`reviews`가 정말 별개 테이블이어야 하는지(예: provider별 평균 별점을 따로 캐싱해야 하는지, 리뷰에 사진이 붙는지 등)도 자연스럽게 드러나서, 스키마 설계가 오히려 더 빨라집니다.

---

위 답변을 아직 안 주셨기 때문에, 지금 가진 정보만으로 일단 스코프 문서의 틀을 잡아봤습니다. 추측한 부분은 "열린 질문"에 표시해뒀으니, 본격적인 데이터 모델링(`data-model-design`) 전에 꼭 확인해주세요.

# 평점/리뷰 — 스코프 (초안, 확인 필요)

## Problem
Requester가 provider를 예약하기 전에 그 provider가 신뢰할 만한지 판단할 근거가 없다. (가정: 이게 핵심 문제라고 추측했습니다 — 실제로 다른 문제, 예를 들어 "provider가 자신의 서비스 품질에 대한 피드백을 못 받는다"가 더 맞다면 알려주세요.)

## Actors
- Requester: provider를 평가하고, 다른 provider를 고를 때 평점/리뷰를 참고하고 싶어한다.
- Provider: 받은 평점과 리뷰를 확인하고, 본인의 신뢰도/평판이 정확히 반영되길 원한다.
- (확인 필요) Admin: 부적절한 리뷰를 검토/삭제하는 역할이 v1에 필요한지 미정.

## User flow (v1, 가정 — 확인 필요)
1. Requester가 provider와의 예약(서비스)을 완료한다.
2. 시스템이 requester에게 평가를 남길 수 있는 상태를 노출한다.
3. Requester가 별점(및 선택적으로 텍스트 리뷰)을 남긴다.
4. 시스템이 provider의 프로필에 해당 평점/리뷰를 반영한다.
5. Provider가 자신에게 달린 리뷰를 확인한다.
   - 5a. 만약 requester가 평가를 남기지 않으면? (자동 만료, 리마인드 여부 미정)
   - 5b. 만약 리뷰 내용이 부적절하면? (신고/삭제 플로우 필요 여부 미정)

## Touches & dependencies
- Entry point in existing product: 예약/서비스 완료 화면 또는 provider 프로필 화면으로 추정 (확인 필요)
- Existing systems it depends on or extends: 예약(booking) 시스템과 연동 필요, 알림 시스템 연동 여부 미정
- New data implied (names only, no schema): `rating`, `review` 엔티티 (그 외 `report`/`reply` 엔티티가 필요한지는 위 질문에 대한 답에 따라 달라짐)

## MVP cut line
**In scope (v1):**
- (가정) Requester가 완료된 예약에 대해 별점 1회 남기기
- (가정) Provider 프로필에 평균 별점 노출

**Out of scope (later):**
- (가정) Provider의 리뷰 답글 기능
- (가정) 리뷰에 대한 "도움이 됐어요" 투표

**Explicitly not doing:**
- (미정 — 확인 필요) 예: 사진/동영상 첨부 리뷰는 영구히 안 할 것인지, 나중에 할 것인지

## Success criteria
- (가정, 확인 필요) "예약 완료 후 7일 내 평점 작성률 OO% 이상" 같은 구체적 수치가 필요. 현재는 placeholder입니다.

## Open questions / assumptions
- 핵심 문제 정의가 추측입니다 — 실제 동기를 알려주시면 스코프가 바뀔 수 있습니다.
- Admin 검수/삭제 플로우가 v1에 필요한지 미정.
- 평가를 안 남기는 경우의 처리(리마인드, 마감 기한)가 미정.
- 리뷰 신고/삭제 플로우가 필요한지 미정.
- 성공 기준 수치가 placeholder이며 실제 목표로 교체 필요.

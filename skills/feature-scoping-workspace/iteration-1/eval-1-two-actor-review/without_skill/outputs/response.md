# 예약 후 상호 리뷰 기능 — 스코프 정리

provider(예약 받는 쪽)와 requester(예약하는 쪽)가 서로 리뷰를 남기는 기능은 단순해 보이지만, "쌍방향"이라는 점 때문에 일반적인 단방향 리뷰(예: 쇼핑몰 상품 리뷰)보다 결정해야 할 게 많습니다. 아래 순서로 정리해봤습니다.

## 1. 먼저 확정해야 할 핵심 질문들

스코프를 잡기 전에 이 질문들에 대한 답이 있어야 디테일이 갈립니다.

1. **리뷰는 양쪽 모두 의무인가, 선택인가?**
   - 한쪽만 작성해도 노출되는지, 양쪽 다 작성해야 노출되는지(블라인드 방식)에 따라 어뷰징 방지 설계가 완전히 달라집니다.
2. **리뷰가 "예약 완료(완료된 거래)"에만 달리는가?**
   - 노쇼, 취소, 거부된 예약에 대해서도 리뷰를 남길 수 있게 할지 결정 필요. (예: "노쇼 신고"는 리뷰와 별개 기능으로 분리하는 게 일반적)
3. **리뷰 대상이 "사람(provider/requester)"인가, "예약 건(서비스/장소)"인가?**
   - 숙박 플랫폼처럼 "이 장소 어땠어요"형 리뷰와, 사람 평판을 쌓는 "이 사람과 거래 어땠어요"형 리뷰는 데이터 모델과 노출 위치가 다릅니다. 둘 다 필요한 경우도 많습니다.
4. **리뷰 공개 시점**
   - 작성 즉시 공개 / 양쪽 다 작성 완료 후 동시 공개(블라인드) / 일정 기간(예: 14일) 후 자동 공개.
5. **별점만? 텍스트도? 태그형 항목도?**
   - provider용과 requester용 평가 항목이 같을 필요는 없습니다. (예: requester는 "친절함, 시간 준수"를, provider는 "정확한 요구사항 전달, 매너"를 평가받는 식으로 비대칭 설계 가능)

## 2. MVP 스코프 (1차 출시 기준 제안)

### 포함 (In Scope)
- **리뷰 작성 대상**: "완료(completed)" 상태인 예약에 한해, provider → requester, requester → provider 양방향 리뷰 작성 가능
- **리뷰 항목**: 별점(1~5점) + 자유 텍스트 코멘트. 태그형 세부 항목은 2차로 미룸
- **작성 기한**: 예약 완료일로부터 N일(예: 14일) 이내. 기한 지나면 작성 불가 처리
- **공개 방식**: 블라인드 방식 — 양쪽 다 작성 완료되거나 기한이 만료되어야 동시 공개 (선의 경쟁/보복성 리뷰 방지)
- **리뷰는 1예약당 1회**: 한 예약에 대해 동일 작성자가 중복 작성 불가, 수정은 가능하되 횟수 제한(예: 공개 전까지만 수정 가능)
- **알림**: 리뷰 작성 요청 알림(완료 직후), 상대방 리뷰 공개 알림
- **프로필 노출**: provider/requester 각각의 프로필에 평균 별점 + 리뷰 개수 + 리뷰 목록 노출
- **신고 기능**: 부적절한 리뷰(욕설, 개인정보, 허위사실) 신고 버튼. 신고 누적 시 운영자 검토 후 숨김 처리

### 제외 (Out of Scope, 2차 이후로 명시적으로 미룸)
- 리뷰에 대한 답글(reply) 기능
- 리뷰 좋아요/도움이 됐어요 투표
- 사진/영상 첨부 리뷰
- 세부 항목별 평가(친절함, 시간 준수 등 다축 평가)
- AI 기반 어뷰징/감정분석 자동 필터링
- 리뷰 기반 정렬/추천 알고리즘 연동
- 리뷰 인센티브(작성 시 포인트 지급 등)

## 3. 데이터 모델 초안

```
Review
- id
- booking_id (FK, 어떤 예약에 대한 리뷰인지 — 중복 작성 방지의 기준)
- author_id (작성자: provider 또는 requester)
- author_role (provider | requester) — 같은 user가 양쪽 역할을 다 할 수도 있으므로 명시 필요
- target_id (리뷰 대상자)
- rating (1~5)
- comment (text, nullable 허용 여부 결정 필요)
- status (pending | published | hidden_by_report | expired)
- created_at
- published_at (블라인드 공개 시점)
- edited_at
```

설계 포인트:
- `booking_id` + `author_role`에 unique 제약을 걸어 중복 작성 방지
- provider/requester가 같은 유저 풀에서 역할만 바뀌는 구조라면(예: 한 유저가 어떤 예약에서는 provider, 다른 예약에서는 requester), 평균 별점은 "역할별로 분리 집계"할지 "통합 집계"할지 결정 필요 — 보통은 역할별 분리가 신뢰도 측면에서 낫습니다.

## 4. 어뷰징/엣지케이스 체크리스트

- **보복성 리뷰**: A가 먼저 안 좋은 리뷰를 받으면 보복으로 B에게 안 좋은 리뷰를 남기는 패턴 → 블라인드 동시공개로 1차 방어
- **노쇼/취소된 예약**: 리뷰 작성 대상에서 제외할지, "노쇼 평가"라는 별도 카테고리로 둘지 결정
- **탈퇴한 유저에 대한 리뷰**: 작성된 리뷰는 유지하되 대상 프로필 링크는 비활성화 처리
- **양쪽 다 작성 안 하고 기한 만료**: 그냥 리뷰 없음으로 종료, 패널티 없음(MVP 기준) — 단, provider 입장에서 평판 쌓기가 막히므로 "리뷰 요청 리마인드 알림" 정도는 넣는 게 좋음
- **동일인이 여러 역할(provider 겸 requester)인 경우**: UI에서 "어떤 역할로 받은 리뷰인지" 명확히 구분 표시
- **법적/약관 이슈**: 허위 리뷰, 명예훼손 관련 신고/삭제 프로세스를 약관에 명시 필요 (운영 정책이지 기능 스코프는 아니지만 출시 전 체크 필요)

## 5. 제안하는 작업 순서

1. 핵심 질문(섹션 1) 답 확정 — 특히 블라인드 공개 여부와 리뷰 대상(사람 vs 거래)
2. 데이터 모델 + API 설계 (리뷰 생성/조회/신고 엔드포인트)
3. "완료" 상태 전이 로직과 연동 (예약 상태 머신에 리뷰 트리거 추가)
4. 알림 연동
5. 프로필 페이지에 평균 별점/리뷰 리스트 노출
6. 신고 처리 운영 플로우(관리자 대시보드 또는 최소한 DB 쿼리 수준 대응)

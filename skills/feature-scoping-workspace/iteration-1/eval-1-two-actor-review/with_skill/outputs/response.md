스코프부터 잡고 가는 게 맞을 것 같아요. 리뷰 기능은 "양쪽이 서로 평가한다"는 말 한 줄 안에 생각보다 많은 결정이 숨어 있어서 (동시에 공개할지, 한쪽만 남겨도 되는지, 안 남기면 어떻게 되는지 등), 화면이나 데이터 구조 얘기로 들어가기 전에 먼저 정리해볼게요.

몇 가지는 일단 합리적으로 가정하고 작성했고, 맨 아래 "열린 질문" 섹션에 표시해뒀습니다. 다르면 거기서 알려주세요.

# 예약 후 상호 리뷰 — 스코프

## Problem
예약이 끝난 뒤, requester는 다른 requester가 이 provider를 예약해도 괜찮을지 판단할 근거가 없고, provider는 어떤 requester가 노쇼나 무리한 요구 없이 예약을 지킬 만한 사람인지 미리 알 방법이 없다. 양쪽 모두 "다음 예약을 결정할 때 참고할 과거 이력"이 없는 상태로 매번 처음 만나는 사람처럼 거래해야 한다.

## Actors
- **Requester**: 예약을 마친 provider에 대해 평가를 남기고, 다른 provider를 고를 때 기존 리뷰를 참고하고 싶어 한다.
- **Provider**: 예약을 마친 requester에 대해 평가를 남기고, 자신에게 달린 리뷰가 다음 requester에게 신뢰 신호로 보이길 원한다.

(두 역할 모두 "리뷰를 남기는 사람"이자 "리뷰를 받는 사람"이라는 점에서 대칭적인 기능입니다.)

## User flow (v1)

1. 예약이 "완료" 상태가 된다 (예: 예약 종료 시각이 지남, 또는 양쪽이 완료를 확인함 — 트리거 정의는 열린 질문 참고).
2. 시스템이 해당 예약과 연결된 두 actor(provider, requester) 모두에게 "리뷰 작성" 알림/진입점을 노출한다.
3. Requester가 해당 예약을 연 provider에 대해 평점 + 코멘트를 작성하고 제출한다.
4. Provider가 해당 예약을 연 requester에 대해 평점 + 코멘트를 작성하고 제출한다.
5. 양쪽 리뷰가 모두 제출되면(또는 일정 기한이 지나면), 리뷰가 상대방 프로필/이력에 공개된다.
   - 5a. 한쪽만 제출하고 다른 쪽이 기한 내 제출하지 않으면: 제출된 리뷰는 기한 종료 후 단독으로 공개된다 (한쪽이 리뷰를 안 남겼다고 다른 쪽 리뷰가 영원히 묻히면 안 됨).
6. 리뷰 대상 actor는 자신에게 달린 리뷰를 프로필에서 확인할 수 있다.
   - 6a. 만약 예약이 취소되었거나 노쇼였다면: 해당 예약에 대해서는 리뷰 작성 진입점을 노출하지 않는다 (완료되지 않은 예약은 리뷰 대상이 아님).
   - 6b. 만약 작성자가 리뷰 작성 기한을 넘기면: 더 이상 그 예약에 대한 리뷰는 남길 수 없다 (기한 정의는 열린 질문 참고).

## Touches & dependencies
- **진입점**: 예약 상세/이력 화면 — "완료된 예약" 목록에 리뷰 작성 액션이 붙는다. Provider/requester 각각의 프로필 화면에는 받은 리뷰 목록이 노출된다.
- **의존하는 기존 시스템**: 예약(booking) 시스템 — 예약의 "완료" 상태 판정에 의존. 알림 시스템 — 리뷰 작성 요청을 양쪽에 보내려면 필요(없다면 v1에서는 진입점 노출만으로 대체 가능).
- **새로 생기는 데이터 (이름만)**: `review` 엔티티 (예약 1건 ↔ 리뷰 최대 2건: provider→requester, requester→provider).

## MVP cut line

**In scope (v1):**
- 완료된 예약 1건당 평점(숫자) + 텍스트 코멘트로 구성된 리뷰 작성 (양방향: provider↔requester)
- 리뷰 작성은 선택사항 (강제 아님)
- 작성된 리뷰는 상대방 프로필에 공개 표시
- 리뷰 작성 기한 존재 (기한 지나면 작성 불가)
- 한쪽만 제출해도 기한 후 단독 공개

**Out of scope (later):**
- 리뷰에 대한 답글(response) 기능
- 리뷰 신고/이의제기 플로우
- 평점 기반 정렬/필터 (검색 결과에 평균 평점 반영)
- 리뷰 수정/삭제
- 사진/첨부파일이 포함된 리뷰

**Explicitly not doing:**
- 리뷰 작성 강제(예약 완료 전까지 다음 예약 차단 등 페널티성 강제)
- AI 기반 리뷰 진위/어뷰징 자동 탐지
- 리뷰에 대한 별도 보상/포인트 지급

## Success criteria
- 예약이 완료 상태가 된 후, provider와 requester 양쪽 모두에게 리뷰 작성 진입점이 노출된다 (수동 확인 가능).
- 한쪽이 리뷰를 제출하면, 다른 쪽이 아직 제출하지 않은 상태에서도 시스템이 기한 이후 단독 공개를 정상적으로 처리한다.
- 작성된 리뷰가 대상 actor의 프로필에서 조회된다.
- 취소/노쇼로 끝난 예약에는 리뷰 작성 진입점이 생성되지 않는다.

## Open questions / assumptions
- **"예약 완료"의 정의를 가정했습니다**: 예약 종료 시각이 지나면 자동으로 완료 처리된다고 가정했습니다. 만약 양쪽의 수동 확인이 필요한 구조라면(예: provider가 "서비스 제공함"을 눌러야 완료), 플로우의 1번 단계가 달라집니다 — 확인해주세요.
- **리뷰 공개 시점(동시 공개 vs 즉시 공개)을 가정했습니다**: "양쪽 다 제출하거나 기한이 지나야 공개"로 가정했습니다(한쪽이 상대 리뷰를 보고 보복성으로 작성하는 것을 막기 위함). 만약 제출 즉시 공개해도 괜찮다고 보신다면 더 단순해집니다 — 확인해주세요.
- **작성 기한 길이는 정하지 않았습니다** (예: 완료 후 14일). 숫자는 비즈니스 판단이 필요해서 비워뒀습니다.
- **평점 스케일(5점? 별점?)은 다음 단계(데이터 모델링/UX)에서 정할 사항으로 남겨뒀습니다.**

좋아, 바로 디자인/구현으로 들어가기 전에 "v1에서 정확히 뭘 만들 것인가"부터 짧게 정리해보자. 아래는 북마크 기능에 대한 스코프 문서야.

# 게시물 북마크 — Scope

## Problem
블로그 독자가 나중에 다시 읽고 싶은 게시물을 발견해도, 이를 저장해둘 방법이 없어 브라우저 탭을 계속 열어두거나, URL을 따로 메모하거나, 결국 다시 찾지 못하고 잊어버린다. 독자가 "지금은 다 못 읽지만 나중에 다시 찾아서 읽고 싶다"는 목표를 달성할 방법이 현재 제품에 없다.

## Actors
이 기능은 단일 사용자 유형(독자/로그인 사용자)만 관련된다. 별도의 운영자/관리자 역할은 v1 범위에서 필요하지 않다.
- 사용자(로그인한 독자): 게시물을 북마크에 추가하고, 나중에 북마크 목록에서 다시 찾아 읽고 싶어한다.

## User flow (v1)
1. 사용자가 게시물 목록 또는 게시물 상세 페이지에서 북마크 아이콘/버튼을 본다.
2. 사용자가 북마크 버튼을 클릭한다.
3. 시스템이 해당 게시물을 사용자의 북마크 목록에 저장하고, 버튼 상태를 "북마크됨"으로 즉시 변경한다.
4. 사용자가 나중에 "내 북마크" 페이지로 이동한다.
5. 시스템이 사용자가 북마크한 게시물 목록을 (최근 북마크한 순으로) 보여준다.
6. 사용자가 목록에서 게시물을 클릭해 원래 게시물 페이지로 이동한다.
7. 사용자가 게시물 상세 페이지 또는 북마크 목록에서 북마크 버튼을 다시 클릭해 북마크를 해제한다.
8. 시스템이 해당 게시물을 북마크 목록에서 제거하고, 버튼 상태를 원래대로 되돌린다.

분기 (예외 상황):
- 2a. 로그인하지 않은 사용자가 북마크 버튼을 클릭하면, 로그인 페이지로 안내하거나 로그인을 유도하는 메시지를 보여준다.
- 5a. 북마크한 게시물이 하나도 없으면, "아직 북마크한 게시물이 없습니다" 같은 빈 상태(empty state)를 보여준다.
- 6a. 북마크한 게시물이 그 사이에 삭제되었거나 비공개로 전환된 경우, 목록에서 해당 항목을 제외하거나 "삭제된 게시물입니다" 표시를 보여준다.

## Touches & dependencies
- Entry point in existing product: 게시물 목록 페이지(카드/리스트 항목)와 게시물 상세 페이지에 북마크 버튼이 추가되고, 별도로 "내 북마크" 페이지(또는 마이페이지 내 탭/섹션)가 새로 생긴다.
- Existing systems it depends on or extends: 인증(로그인 여부 확인 및 사용자 식별)에 의존한다. 결제, 알림, 파일 스토리지는 관련 없음.
- New data implied (names only, no schema): `bookmark` (사용자–게시물 간의 관계를 나타내는 엔티티)

## MVP cut line
**In scope (v1):**
- 게시물에 북마크 추가/해제 (토글)
- "내 북마크" 목록 페이지 (북마크한 게시물 보기)
- 로그인 사용자만 사용 가능

**Out of scope (later):**
- 북마크를 폴더/태그로 분류하거나 정리하는 기능
- 북마크 목록 정렬/필터 옵션 (날짜 외 다른 기준)
- 북마크한 게시물에 새 댓글이 달리면 알려주는 알림
- 북마크를 다른 사용자와 공유하는 기능

**Explicitly not doing:**
- 비로그인(익명) 사용자를 위한 로컬/브라우저 기반 북마크 저장
- 북마크 수에 대한 제한(쿼터) 설계
- 북마크 데이터를 다른 서비스로 내보내기/가져오기(export/import)

## Success criteria
- 로그인한 사용자가 게시물 상세 또는 목록 페이지에서 클릭 1번으로 북마크를 추가/해제할 수 있다.
- 사용자가 "내 북마크" 페이지에 진입했을 때, 자신이 북마크한 게시물이 빠짐없이, 그리고 북마크하지 않은 게시물은 섞이지 않고 표시된다.
- 동일한 게시물을 두 번 북마크해도 목록에 중복으로 나타나지 않는다.
- 북마크 추가/해제 동작이 페이지를 새로고침하지 않고도 화면에 즉시 반영된다.

## Open questions / assumptions
- "내 북마크" 페이지가 별도의 새 페이지인지, 기존 "마이페이지/프로필" 안의 한 섹션인지 가정하지 않고 확인이 필요함 (현재는 별도 또는 통합 둘 다 가능하다고 가정하고 진행).
- 비공개(작성자만 볼 수 있는) 게시물이 이 블로그 서비스에 존재하는지 불확실함 — 존재한다면 비공개 게시물의 북마크 노출 정책을 추가로 정해야 함.
- 게시물이 삭제된 경우 북마크 레코드를 어떻게 처리할지(완전 삭제 vs. 표시만 숨김)는 데이터 모델링 단계에서 결정 필요.
- 한 사용자가 북마크할 수 있는 게시물 수에 상한이 있는지는 현재 정하지 않음 — 필요성이 보이면 이후 단계에서 논의.

---
이 정도면 "북마크가 정확히 뭘 하는 기능인지"는 분명해졌다고 생각해. 위에서 가정한 부분(특히 "내 북마크" 페이지 위치, 비공개 게시물 처리)은 한번 확인해주면 좋겠고, 확인되면 이 문서를 가지고 다음 단계(화면 흐름 설계 → 데이터 모델링)로 넘어가면 될 것 같아.

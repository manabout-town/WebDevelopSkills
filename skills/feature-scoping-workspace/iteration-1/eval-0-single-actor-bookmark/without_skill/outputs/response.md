# 블로그 게시물 북마크 기능 설계

블로그에 "북마크(저장)" 기능을 추가하는 거군요. 큰 기능은 아니지만 데이터 모델, API, UX를 한 번에 잘 잡아두면 나중에 "내가 저장한 글" 페이지나 인기 게시물 집계 같은 것으로도 확장하기 쉬워요. 단계별로 정리해봤습니다.

## 1. 요구사항부터 명확히 하기

먼저 결정해야 할 것들이에요. (정답이 없는 부분이니 서비스 성격에 맞게 고르면 됩니다.)

- **누가 북마크할 수 있나?** 로그인한 사용자만 가능하게 할 가능성이 높음 (비로그인 사용자는 로그인 유도)
- **북마크 목록을 다른 사람도 볼 수 있나?** 보통은 본인만 보는 비공개 기능
- **북마크에 폴더/태그 같은 분류가 필요한가?** 처음엔 없이 단순 "저장/저장 취소"로 시작하는 걸 추천
- **알림이나 통계에 연동되나?** 예: "이 글은 N명이 북마크함" 같은 카운트 노출 여부

일단 가장 단순한 버전(로그인 사용자가 게시물을 저장/취소하고, 본인 목록만 보는 것)을 기준으로 설계를 잡고, 이후 확장은 별도로 얹는 방식을 추천합니다.

## 2. 데이터 모델

가장 핵심은 **사용자-게시물 간 다대다 관계 테이블**을 만드는 것입니다.

```sql
CREATE TABLE bookmarks (
  id          BIGSERIAL PRIMARY KEY,
  user_id     BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  post_id     BIGINT NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (user_id, post_id)
);

CREATE INDEX idx_bookmarks_user_id ON bookmarks(user_id, created_at DESC);
CREATE INDEX idx_bookmarks_post_id ON bookmarks(post_id);
```

포인트:
- `UNIQUE (user_id, post_id)`로 중복 북마크를 DB 레벨에서 막기
- `user_id` 기준 인덱스는 "내 북마크 목록" 조회용, `post_id` 기준은 "이 글의 북마크 수" 집계용
- `created_at`으로 정렬해서 최신 북마크 순으로 보여줄 수 있음
- 게시물이나 유저가 삭제되면 북마크도 같이 정리되도록 `ON DELETE CASCADE` 설정

별도 테이블로 분리하는 이유는, `posts`나 `users` 테이블에 배열 컬럼으로 넣는 방식보다 조회/정합성/확장성이 훨씬 좋기 때문이에요 (특히 "이 글을 북마크한 사람 수" 같은 집계가 쉬워짐).

## 3. API 설계

REST 기준으로 예시를 들면:

| 메서드 | 엔드포인트 | 설명 |
|---|---|---|
| POST | `/api/posts/:postId/bookmark` | 북마크 추가 |
| DELETE | `/api/posts/:postId/bookmark` | 북마크 취소 |
| GET | `/api/me/bookmarks` | 내 북마크 목록 (페이지네이션) |
| GET | `/api/posts/:postId` 응답에 `isBookmarked: boolean` 포함 | 게시물 상세/목록 조회 시 현재 사용자의 북마크 여부 같이 내려주기 |

설계 팁:
- 토글 형태(POST/DELETE)보다 `PUT`으로 idempotent하게 처리하거나, 클라이언트에서 현재 상태를 보고 분기해도 됩니다. 다만 "이미 북마크된 글에 또 북마크 요청"이 와도 에러 없이 200을 주는 게 UX상 안전합니다.
- 게시물 목록(피드)을 조회할 때마다 N개의 게시물에 대해 매번 북마크 여부를 조회하면 N+1 문제가 생기니, 목록 API에서 한 번에 `IN` 쿼리로 조회해서 `isBookmarked` 필드를 같이 내려주는 걸 추천합니다.

## 4. 프론트엔드 UX

- 게시물 카드/상세 페이지에 북마크 아이콘(별표, 책갈피 모양 등) 토글 버튼
- 클릭 즉시 UI를 먼저 바꾸는 **낙관적 업데이트(optimistic update)** 적용 → 클릭 반응이 즉각적으로 느껴짐, 실패 시 롤백 처리
- 비로그인 상태에서 클릭하면 로그인 페이지로 유도 (또는 로그인 모달)
- "내가 북마크한 글" 마이페이지/전용 탭 추가
  - 빈 상태(북마크한 글이 없을 때) 안내 문구도 챙기기
  - 페이지네이션 또는 무한 스크롤

## 5. 고려해야 할 엣지 케이스

- 이미 삭제된 게시물이 북마크 목록에 남아있는 경우 처리 (CASCADE로 자동 삭제되게 했으면 문제 없음)
- 비공개 처리된/임시저장 게시물을 북마크했다가 작성자가 비공개로 바꾼 경우 목록에서 어떻게 보여줄지
- 같은 사용자가 빠르게 두 번 클릭(더블 클릭, 네트워크 지연 중 재클릭)했을 때 중복 요청 방어 — DB unique 제약 + 클라이언트 디바운스로 방어
- 북마크 수가 많아질 경우 "이 글을 북마크한 사람 수" 집계를 실시간 COUNT 쿼리로 할지, 캐시/비정규화 컬럼(`posts.bookmark_count`)으로 관리할지 결정 (트래픽이 적다면 처음엔 COUNT 쿼리로 충분)

## 6. 구현 순서 제안

1. DB 마이그레이션으로 `bookmarks` 테이블 추가
2. 북마크 추가/취소 API 구현 (인증 미들웨어 적용)
3. 게시물 조회 API에 `isBookmarked` 필드 추가
4. 프론트엔드 북마크 버튼 컴포넌트 + 낙관적 업데이트
5. "내 북마크" 페이지 추가
6. (선택) 북마크 수 표시, 인기순 정렬, 폴더/태그 분류 등 확장 기능

## 정리

핵심은 user-post 다대다 관계 테이블 하나로 거의 다 해결된다는 점이에요. 처음부터 폴더 분류나 공유 기능까지 욕심내지 말고, "저장/취소 + 내 목록 보기" 최소 버전부터 만들고 사용자 반응 보면서 확장하는 게 좋습니다.

추가로 궁금한 부분 있으면 말씀해주세요 (예: 사용 중인 프레임워크/DB가 정해져 있다면 그에 맞춘 구체적인 코드도 같이 짜드릴 수 있어요).
